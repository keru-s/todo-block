# 由各自规则形成变化，由统一操作单元完整应用

每一种用户规则只有一个明确归属：父子组移动、结转、粘贴、文字编辑及其他规则分别在各自模块中决定用户最终应得到的状态，不在界面、存储或其他入口重复实现，也不把全部规则集中到一个控制中心。所有规则形成的变化都交给同一个操作单元模块，由它统一判断是否有变化、检查能否完整应用、保持待办身份和相关选择与焦点、执行变化并登记撤销与恢复；这样既保留规则本身的 locality，又让完整应用和操作历史获得共同保障。

各规则只提交受影响用户状态的完整前后结果，不提交“先删除、再插入、再改顺序”之类的执行步骤。新建、删除、普通修改、完成状态和选择变化都表达为结果之间的转换；撤销与恢复只应用当时记录的另一侧结果，不重新运行原规则，因此规则以后改变也不会改变已有操作单元的含义。

所有待办变化使用同一种前后状态表达：某一侧可以不存在，另一侧是包含待办身份在内的完整状态。新建是“不存在到存在”，删除是“存在到不存在”，文字、完成、层级、归属和顺序变化都是“存在到另一份存在状态”。选择与编辑焦点作为列表用户状态单独记录，不再为完成状态、新建删除和普通修改保留彼此不同的操作记录分类。

普通操作先形成完整变化，再由操作单元模块应用；连续输入段为了保持即时输入体验，可以先让文字生效，并在输入段结束时把已经发生的完整前后状态交给同一模块登记。两种进入方式共享相同的有效性、历史与恢复规则，不形成两套操作历史。

操作单元在当前运行中的用户状态和操作历史完整一致后即视为成功，不为每次输入同步等待本地存储。正常的合并写入等待属于保存排队，不提示用户；只有实际写入失败才进入待保存状态。已经显示并进入历史的结果不得被静默撤回，应用保留当前准确状态并自动重试、持续明确提示用户。待保存期间继续允许编辑、撤销和恢复，后续保存只写入当时的最终状态；退出时仍未保存则阻止无提示退出。

操作单元模块只产出撤销或恢复后的结果说明，包括受影响待办、结果所在列表以及应恢复的选择与编辑焦点，不直接打开窗口、切换页面或滚动界面。外层界面负责把结果展示给用户；菜单栏仍只能处理历史中下一步且结果可由菜单栏列表展示的操作，不能筛选或跳过其他步骤。

每个操作单元只额外记录一个操作关注点，即用户意图主要围绕的待办身份或列表。结果说明优先从目标侧真实的编辑焦点和选择推导，其次使用操作关注点；关注待办在目标侧不存在时可以从另一侧记录确定原列表。规则不分别提交前后展示结果，统一模块也不依赖变化集合的排列顺序猜测。只有出现现有用户状态和单一操作关注点确实无法表达的新行为时，才重新评估是否扩展。

当前列表只向操作单元模块声明自己能展示全部结果或仅能展示今天结果。撤销与恢复的可用判断和真正执行都由操作单元模块检查同一条历史下一步，执行时再次复核日期、当前状态和展示范围；外层不能读取历史内容自行筛选。复核不通过时历史保持原位，并返回未执行原因。

选择与编辑焦点以状态值及稳定归属记录，不依赖某个仍然存活的界面对象。从不同入口撤销或恢复时，原操作列表恢复其记录状态，当前负责展示结果的列表也定位到该结果；其他列表保留各自状态。原界面消失后重新创建，仍能根据记录恢复。

操作历史由应用直接保存结构化操作单元，不再借用系统撤销机制充当隐藏栈。应用已经完全接管撤销与恢复菜单、快捷键和文字输入历史，系统机制没有承担界面集成，却迫使“是否影响今天”等信息维护为另一份平行记录。替换后撤销栈、恢复栈、最多 50 步、下一步结果范围和过期记录处理都从同一份操作单元得出；迁移必须先用等价性测试证明现有行为全部保留，再删除旧实现。

所有会进入操作历史的用户状态变化只能由操作单元模块真正写入。规则模块只计算完整前后结果，不能先修改待办再补写历史；连续输入段同样通过操作单元模块即时应用文字，只在输入段结束时完成历史登记。操作引发的选择和编辑焦点变化随操作单元一起应用；纯导航不进入历史，可以独立改变选择、焦点和光标。

规则模块只能读取不可由其修改的当前状态，并返回完整结果，不持有可写待办或存储。操作单元模块在应用前重新确认当前状态与结果记录的前态一致；任何前态过期都会让整个动作不生效，并按未执行动作处理，不能把部分新结果写入较新的用户状态。

## Consequences

- 同一条用户规则不得在多个生产入口重复实现；界面和存储只转交意图或执行统一结果。
- 操作单元记录结果，不记录步骤；调用方不能把执行顺序作为正确性的一部分泄漏给统一模块。
- 所有待办变化共享一种前后状态表达和一套检查、应用与恢复规则。
- 操作单元模块是历史性用户状态变化的唯一写入口；生产代码不得绕过它直接修改后再补登记。
- 规则结果必须以其读取的完整前态为应用条件；前态过期时整步拒绝，不尝试合并或尽力应用。
- 规则测试只验证该规则产生的用户状态，操作单元测试统一验证完整应用、待办身份、选择与焦点、撤销和恢复。
- 新增用户规则时必须先确定唯一归属，再接入统一操作单元；不能直接拼接另一套撤销登记。
- 本地存储是操作单元之后的可靠保存步骤；保存失败不能改写该操作单元是否成功，只能进入待保存状态。
- 结果计算属于操作单元模块，窗口和导航属于外层界面；两者通过结果说明衔接。
- 操作关注点只补充用户意图，不复制列表、选择、焦点或导航信息；结果说明必须从已记录的真实状态推导。
- 当前列表只声明展示范围，操作单元模块原子地复核并执行历史下一步，任何入口都不能跳过或自行筛选历史。
- 选择与编辑焦点记录不持有界面对象，恢复行为不取决于原界面是否仍然存活。
- 成功撤销后同一操作单元才从撤销方向进入恢复方向，成功恢复时严格对称；无变化或未执行动作不得移动历史。
- 撤销后只有新的成功操作才能清除恢复路径；纯导航、复制、保存重试和其他无变化动作都不能清除。
- 过期操作单元整条丢弃后继续寻找下一条可完整应用记录，历史上限保持 50 步。
- 替换历史实现前必须建立新旧共用的行为对照测试，并覆盖连续撤销恢复、撤销后新操作、无变化动作、过期记录和跨列表混合序列。
- 必须用自动生成的操作序列逐步比对一个独立的小型参考历史，核对用户状态、待办身份和两个历史方向；中文输入法、语音输入与菜单栏切换继续实际验收。
- 只有新旧结果完全一致，旧系统历史实现才可删除；任何行为差异都必须先解决，不能作为迁移例外保留。
- 迁移按统一前后状态入口、规则只读化与旁路清理、自有结构化历史、保存失败恢复依次推进；每一步独立验证后再继续。
- 迁移期间生产环境始终只有一条正式操作历史，不同时写入新旧历史。现有变化到统一前后状态的临时转换在全部入口迁移后立即删除，不形成长期双轨。
