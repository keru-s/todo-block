# 列表动作模块与命令接管计划

## 目标

把“深化编辑器动作模块”和“让当前列表真正拥有命令接管”合并为一次改造。每个可见待办列表拥有一份稳定的动作模块，列表内部动作与应用菜单命令都从这里进入；应用在任何时刻只把命令交给一个当前列表。

最终只保留新入口，不在产品中维护新旧双轨。

## 已确认的产品规则

### 当前列表

- 列表中的直接用户动作先令该列表接管，再执行动作。
- 菜单栏面板打开后立即由菜单栏列表接管，无需先点击待办。
- 菜单栏关闭后恢复仍然存在的原列表；原列表已消失时保持无接管状态，等待另一列表主动接管。
- 主窗口、长期列表和菜单栏分别保留自己的选择、文字光标和多选范围，接管不会覆盖其他列表的状态。
- 延迟保存、输入段归档和界面刷新不能抢回接管权。
- 用户动作主动展示的新列表可以接管；纯后台工作不能接管。

### 模块边界

- 模块负责待办文字、选择、新建、删除、完成、层级、移动、拖拽、日期、复制粘贴、撤销恢复、命令可用状态和未执行原因。
- 窗口、侧边栏导航、菜单栏面板开关和应用设置继续由外层界面负责。
- 菜单与快捷键使用一组统一的应用级命令；文字、拖拽等需要不同信息的编辑器动作保留各自清晰的专用形式。
- 不把全部动作强塞进一种巨型通用命令。

### 文字与输入法

- 文字光标或文字选区位于标题中时，文字类命令优先。
- 有文字选区时，复制和剪切只处理文字；只有光标但没有文字选区时，复制和剪切不可用，不能降级为整条待办操作。
- 文字编辑中粘贴会插入光标位置，不会新增待办。
- 普通中文输入的一次完整选词属于同一连续输入段。
- 一次语音输入从开始到明确结束属于同一连续输入段；停顿和系统对既有识别文字的动态改写不会拆段。
- 临时文字尚未确认时，键盘结构命令先交给输入法，不直接修改待办。
- 用户点击标题之外的明确动作时，先结束并保存完整输入段，再执行下一动作。

### 移动

- 任何被移动待办都带上它的全部后代，完成状态不改变父子组边界。
- 文字编辑中的 Command+上/下移动光标所在父子组，并保留现有层级与归属变化规则。
- 普通列表多选状态下，应用菜单的上移/下移处理全部选中的父子组，并去掉父项与后代的重复选择。

### 操作历史

- 全应用继续只有一条严格按时间排列的操作历史，不按列表拆分，也不能跳过中间步骤。
- 主窗口可以执行全部撤销与恢复；结果属于其他月份或长期列表时，自动切换并展示结果，新展示列表随后接管。
- 菜单栏只展示和操作今天的待办，不支持跨到未来月份或长期列表。
- 菜单栏可以撤销或恢复任何直接影响今天列表的历史下一步，无论该操作最初来自菜单栏还是主窗口。
- 历史下一步完全不涉及今天时，菜单栏中的撤销或恢复不可用，快捷键尝试时提示用户前往主窗口；不能跳过这一步寻找更早的今天操作。

### 未执行反馈

- 菜单项根据当前列表和文字状态显示真实可用性。
- 可见边界安静处理，例如已在最上方仍尝试上移。
- 用户看不见的拒绝原因在当前列表内部底部显示短暂提示，例如“请先结束当前输入”或“请在主窗口撤销上一次操作”。
- 提示采用已确认的深色半透明圆角样式，带信息图标、无按钮、不抢焦点、数秒后消失；重复提示替换原提示，不叠加。
- 复制、剪切及其他日常成功操作不显示确认提示。
- 只有不可逆且可能造成意外数据丢失的操作才使用弹窗。

## 设计形状

### 每个列表一份稳定模块

每个可见列表创建一次自己的动作模块，模块持有该列表的范围与选择状态，并向编辑器提供专用动作，向应用菜单提供统一命令能力。界面刷新不能反复创建模块，否则会丢失接管身份和延迟输入状态。

### 单一接管协调者

应用只保留一个接管协调者，用来：

- 保存当前列表的有效身份；
- 处理菜单栏的临时接管与恢复；
- 转发应用级命令和可用状态；
- 拒绝已经消失的列表；
- 保证延迟回调不会改变当前列表。

协调者不保存第二份选择，也不执行待办规则。

### 明确的执行结果

每个应用级命令必须返回以下结果之一：

- 已执行并改变用户状态；
- 当前边界下无变化；
- 因隐藏条件未执行，并附带可展示原因。

菜单可用性与实际执行使用同一判断，避免菜单显示可用但执行后无反应。

### 外层展示职责

动作模块可以说明撤销或恢复结果应展示在哪里，但打开窗口和切换侧边栏仍由外层界面完成。当前列表内部的短暂提示由编辑器列表表面展示。

## 迁移步骤

1. 建立稳定的列表动作模块、统一应用级命令和执行结果。
2. 建立单一接管协调者，覆盖普通接管、菜单栏临时接管、恢复和列表失效。
3. 把现有编辑器动作接入模块，保持文字、拖拽和层级规则不变。
4. 接入文字优先、输入法临时文字和语音动态改写的连续输入规则。
5. 把应用菜单的复制、剪切、粘贴、上移、下移、撤销和恢复改为只查询当前列表模块。
6. 接入主窗口、长期列表和菜单栏；列表外按钮也通过所属列表模块执行待办动作。
7. 加入列表内底部短暂提示和辅助功能朗读。
8. 删除旧动作工厂、独立剪贴板接管、独立移动接管，以及历史展示中保存当前范围的旧入口。
9. 扫描所有待办写入与命令入口，确认没有绕过模块的界面路径。

## 自动检查

- 当前列表接管、临时覆盖、恢复、列表失效和延迟回调不抢占。
- 三个列表实例各自保留选择和文字光标。
- 应用菜单的可用状态与执行结果一致。
- 文字选择、仅有光标、整条待办选择下的复制、剪切和粘贴。
- 普通输入、中文输入法组合、语音动态替换、停顿和明确结束后的单步撤销恢复。
- 临时文字期间的结构命令与点击外部动作顺序。
- 局部父子组移动与多选父子组移动。
- 菜单栏今天历史、主窗口今天历史、未来月份和长期历史的撤销恢复边界。
- 未执行原因提示不进入操作历史、不抢焦点、不重复堆叠。
- 原有存储、排序、层级、拖拽、剪贴板和全局历史检查全部继续通过。

## 实际验收

- 主窗口日期列表、长期列表和菜单栏分别执行文字、选择、新建、删除、完成、层级、移动、复制粘贴、撤销和恢复。
- 主窗口与菜单栏同时保留不同选择，打开和关闭菜单栏后命令目标正确恢复。
- 使用中文输入法和 macOS 语音输入，观察动态修正文字并验证一次撤销恢复完整会话。
- 在临时输入期间尝试结构快捷键和外部点击，确认没有丢字、乱序或部分撤销。
- 验证列表底部提示样式、消失时机、键盘焦点和辅助功能朗读。
- 重新启动应用，确认待办结果保持一致，当前列表与操作历史按运行期规则重新开始。

## 完成标准

- 三个列表入口全部使用新模块和同一接管协调者。
- 产品中不再存在第二套当前列表回调或旧动作入口。
- 所有自动检查通过，应用构建成功。
- 上述实际验收全部通过，特别是中文输入法、语音输入、菜单栏历史限制和接管恢复。
