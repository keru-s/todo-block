# AppKit 编辑器重构计划

这份文档只用于审阅计划。当前分支不开始实际重构，不修改现有功能代码。

## 1. 结论

长期看，可以把高频使用的待办编辑区域迁移到 AppKit。这里的“编辑区域”主要指：

- 待办项文本编辑
- 键盘移动焦点
- 缩进和取消缩进
- 拖拽排序
- 跨日期移动
- 长期待办分桶移动
- 多选
- 复制粘贴
- 撤销重做后的焦点恢复

不建议第一步就把整个 app 全量改成 AppKit。更稳的路线是：

1. 保留现在的数据模型、存储、排序、撤销、剪贴板等核心逻辑。
2. 新建一个 AppKit 版本的待办编辑器。
3. 先把主窗口里的待办列表编辑区域替换掉。
4. 验证稳定后，再决定菜单栏列表、长期列表、侧边栏是否继续迁移。

这样既能解决当前最常修、最容易出问题的交互区域，也不会一次性重写所有稳定部分。

## 2. 为什么值得做

这个 app 的核心价值不是“展示待办”，而是“快速编辑待办”。最常用、也最容易出问题的地方，正好是 AppKit 更擅长的部分：

- 文本输入和输入法组合状态
- 光标和焦点控制
- 键盘命令
- 鼠标拖拽
- 多选
- 列表内排序
- 跨列表移动
- macOS 原生菜单和响应链

当前代码已经说明 SwiftUI 在这些地方不够省心：

- 文本输入已经包了一层 `NSTextView`
- 菜单栏已经用了 `NSStatusItem` 和 `NSPopover`
- Option 拖选已经用了 `NSEvent`
- 拖拽落点需要手动记录全局位置
- 很多修复都集中在待办项移动、层级、拖拽和焦点恢复上

也就是说，现在不是“纯 SwiftUI 简单界面”，而是“SwiftUI 外壳 + 很多 AppKit 补丁”。如果后续还要持续强化编辑体验，把编辑器主体迁到 AppKit 是合理的长期方向。

## 3. 目标

- 让待办项编辑、拖拽、层级调整、跨日期移动更稳定。
- 降低 SwiftUI 和 AppKit 混用边界带来的问题。
- 保留当前已经验证过的核心逻辑，避免重写业务规则。
- 每一步都能单独验证，不做一次性大爆炸重写。
- 迁移过程中保持 app 可运行、可回退、可对比。
- 让后续新增复杂编辑功能更容易维护。

## 4. 不做什么

- 不换 SwiftData。
- 不引入第三方库。
- 不重写数据存储。
- 不重写排序、撤销、剪贴板等已经有测试保护的核心逻辑。
- 不在第一阶段改变用户可见行为。
- 不一开始就全量重写整个 app。

## 5. 当前结构判断

### 5.1 适合暂时保留 SwiftUI 的部分

这些地方主要是页面组织和信息展示，当前没有明显必要马上重写：

- app 入口
- 主窗口外壳
- 侧边栏
- 主窗口左右分栏
- 月份和长期列表的导航结构
- 部分静态空状态和标题区域

对应文件主要包括：

- `todo_blockApp`
- `ContentView`
- `SidebarView`
- `MainDetailHostView`
- `TodoListView`
- `LongTermListView`

### 5.2 最适合迁移到 AppKit 的部分

这些地方是高频编辑核心，也是之前反复修复最多的区域：

- `TodoDroppableList`
- `TodoItemView`
- `CustomTextEditor`
- `CustomNSTextView`
- `DaySectionView` 里的待办项列表
- `LongTermBucketView` 里的待办项列表
- `TodoDropIndicatorOverlay`
- `OptionDragSelectionMonitor`
- `TodoDragCoordinator`

这里应该成为第一轮重构的核心目标。

### 5.3 应该保留的核心逻辑

以下逻辑已经比较独立，不应该因为 UI 改成 AppKit 就重写：

- `TodoStore`
- `SelectionManager`
- `TodoReorderMoveEngine`
- `TodoDropLocationEngine`
- `TodoKeyboardReorderEngine`
- `TodoReorderCommandManager`
- `TodoClipboardManager`
- `TodoUndoManager`
- `MarkdownTodoCodec`

如果迁移中发现这些逻辑需要调整，也应该把它当作“接口补强”，而不是顺手重写。

## 6. 目标架构

### 6.1 新增一个 AppKit 编辑器模块

建议新建目录：

```text
todo block/
├── Views/
│   └── AppKitEditor/
│       ├── TodoEditorRepresentable.swift
│       ├── TodoEditorViewController.swift
│       ├── TodoEditorScrollView.swift
│       ├── TodoEditorListView.swift
│       ├── TodoEditorRowView.swift
│       ├── TodoEditorTextView.swift
│       ├── TodoEditorDragController.swift
│       ├── TodoEditorSelectionController.swift
│       └── TodoEditorLayout.swift
```

实际文件名可以在实现时调整，但边界要清楚：

- SwiftUI 负责页面外壳和把目标列表传给编辑器。
- AppKit 负责待办列表内部的渲染、文本输入、焦点、鼠标、拖拽、选择、键盘命令。
- 数据变化仍然通过现有的 store 和服务层完成。

### 6.2 和 SwiftUI 的连接方式

建议用一个桥接视图把 AppKit 编辑器嵌回当前 SwiftUI 页面。

第一选择是控制器级桥接，因为编辑器会涉及：

- 滚动区域
- 文本焦点
- 鼠标事件
- 拖拽生命周期
- 响应链
- 菜单命令

这样 SwiftUI 继续负责页面组合，AppKit 专注处理复杂编辑。

### 6.3 数据流

不要让 AppKit 编辑器维护一份独立的永久数据。

推荐流程：

1. SwiftUI 页面决定当前展示哪个列表。
2. AppKit 编辑器从 `TodoStore` 获取对应待办项。
3. 编辑器渲染这些待办项。
4. 用户操作触发现有 store 或服务方法。
5. store 变更后通知刷新。
6. AppKit 编辑器只刷新必要的行，并尽量保留焦点和选择状态。

允许 AppKit 内部保存临时状态，比如：

- 当前焦点
- 当前选择
- 当前拖拽会话
- 每一行的位置
- 输入法组合状态
- 光标位置

但待办数据本身不能出现两份来源。

### 6.4 行状态模型

建议给 AppKit 行渲染定义一个轻量状态：

```text
TodoEditorRowState
- item id
- 标题
- 是否完成
- 缩进层级
- 是否选中
- 是否聚焦
- 是否正在输入法组合
```

这样行视图只负责显示和事件转发，不直接拥有业务规则。

## 7. 分阶段计划

### 阶段 0：先锁定当前行为

目的：在改 UI 前，先确认哪些行为必须保持一致。

要做的事：

- 整理当前编辑行为清单。
- 补齐必要的核心逻辑测试。
- 明确 AppKit 编辑器要调用哪些现有接口。
- 确认哪些行为必须完全保留，哪些可以趁迁移优化。

需要锁定的行为：

- Enter 在不同光标位置创建待办。
- Backspace 删除空待办。
- Tab 和 Shift+Tab 修改层级。
- 上下箭头在待办之间移动焦点。
- Command+Up 和 Command+Down 移动待办块。
- 拖拽时父子块不被拆散。
- 跨日期移动时整块移动。
- 拖到长期分桶时行为正确。
- 多选后删除、复制、移动正确。
- 复制粘贴保留层级。
- 撤销重做后数据、顺序、焦点尽量恢复正确。

完成标准：

- 有一份当前行为清单。
- 当前核心测试通过。
- 缺失的关键测试已经补上。
- 还没有替换现有界面。

### 阶段 1：建立 AppKit 编辑器骨架

目的：先证明 AppKit 编辑器能嵌入当前 app，不碰复杂交互。

要做的事：

- 新建 `Views/AppKitEditor/`。
- 新建 AppKit 编辑器入口。
- 先只展示一个只读列表。
- 从 `TodoStore` 读取当前日期或长期列表数据。
- 加一个内部开关或预览入口，用来和现有 SwiftUI 列表对比。

完成标准：

- app 能构建。
- AppKit 编辑器能显示同一批待办。
- 当前正式界面行为不变。
- 不引入编辑、拖拽、多选逻辑。

### 阶段 2：迁移文本编辑和键盘操作

目的：把最核心的输入体验迁到 AppKit。

要做的事：

- 实现待办行。
- 实现复选框、拖拽手柄、文本输入区域。
- 直接使用 AppKit 文本控件处理输入。
- 接管 Enter、Backspace、Tab、Shift+Tab、上下箭头、Space、Command+Up、Command+Down。
- 保留输入法组合状态。
- 保留创建、删除、撤销、重做后的焦点恢复。

完成标准：

- 打字、换行、拆分、新建待办体验正常。
- 键盘优先流程顺畅。
- 中文输入法不出问题。
- 长文本自动换行后行高正确。
- 当前核心测试通过。

### 阶段 3：迁移选择能力

目的：把单选、范围选择、拖选都交给 AppKit。

要做的事：

- 实现点击选择。
- 实现 Shift 点击范围选择。
- 实现 Option 拖选。
- 不再依赖 SwiftUI 全局位置偏移来判断拖选区域。
- 保持选择状态能被复制、删除、移动命令使用。

完成标准：

- 日期列表选择正常。
- 长期列表选择正常。
- 多选后复制、删除、移动目标正确。
- 拖选不影响普通文本选择。
- 当前选择相关测试通过。

### 阶段 4：迁移列表内拖拽排序

目的：先解决单个列表内部拖拽排序。

要做的事：

- 用 AppKit 跟踪拖拽开始、移动、结束。
- 用 AppKit 行位置计算插入点。
- 用 AppKit 绘制插入提示线。
- 最终移动仍然调用现有排序逻辑。
- 保留父子块整体移动。

完成标准：

- 同一天内拖拽排序正常。
- 长期分桶内拖拽排序正常。
- 父项带子项时不会被拆散。
- 插入提示线稳定，不残留。
- 当前排序相关测试通过。

### 阶段 5：迁移跨目标移动

目的：支持跨日期、跨长期分桶移动。

要做的事：

- 定义日期区域、长期分桶、侧边栏的落点。
- 尽量减少现有 SwiftUI 全局坐标桥接。
- 跨目标移动继续调用现有 store 方法。
- 明确跨目标移动时是否重置缩进。
- 明确移动到月份时落到哪个日期。

完成标准：

- 日期到日期移动正常。
- 日期到长期分桶移动正常。
- 长期分桶回日期移动正常。
- 父项带子项跨目标移动仍然保持整体。
- 取消拖拽不会改变数据。
- 不残留插入提示线。

### 阶段 6：替换主窗口编辑区

目的：让主窗口正式使用 AppKit 编辑器。

要做的事：

- 替换日期分区里的待办列表。
- 替换长期分桶里的待办列表。
- 保留 SwiftUI 侧边栏和页面外壳。
- 保留旧 SwiftUI 编辑器一段时间，方便对照和回退。
- 稳定后再删除不用的旧列表代码。

完成标准：

- 主窗口待办编辑区使用 AppKit。
- 日期列表和长期列表都能正常编辑。
- 现有主要功能没有退化。
- 手工验收清单通过。

### 阶段 7：决定菜单栏是否迁移

目的：主窗口稳定后，再决定菜单栏列表怎么处理。

要做的事：

- 对比菜单栏列表和主窗口列表的行为差异。
- 判断是否复用同一个 AppKit 编辑器。
- 如果菜单栏继续使用 SwiftUI，要记录原因。
- 如果迁移菜单栏，要确保弹窗生命周期稳定。

完成标准：

- 菜单栏列表有明确长期方案。
- 如果迁移，行为尽量和主窗口一致。
- 如果不迁移，重复逻辑是有意保留，不是遗漏。

### 阶段 8：清理和文档更新

目的：删除旧代码，更新项目说明。

要做的事：

- 删除不用的 SwiftUI 编辑器代码。
- 删除不用的拖拽桥接代码。
- 更新 README 技术说明。
- 更新项目协作说明。
- 保留仍然必要的 SwiftUI/AppKit 桥接。

完成标准：

- 没有明显死代码。
- 文档和真实架构一致。
- 构建和测试通过。

## 8. 手工验收清单

每个会改变编辑行为的阶段，都应该跑这份清单。

### 基础编辑

- 新建待办。
- 在文本中间按 Enter 拆分。
- 在文本开头按 Enter 创建上方待办。
- 删除空待办。
- 切换完成状态。
- 编辑长文本并自动换行。
- 使用中文输入法输入和选词。

### 键盘操作

- 上下移动焦点。
- 在多行待办之间移动焦点。
- 保持横向光标位置。
- Tab 增加缩进。
- Shift+Tab 减少缩进。
- Command+Up 移动父项。
- Command+Down 移动父项。
- 对子项执行 Command+Up 和 Command+Down。

### 层级

- 缩进到最大层级。
- 从最大层级逐级退回根层级。
- 移动带子项的父项。
- 移动带孙项的子项。
- 确认父子块不会被拆开。

### 拖拽

- 同一天内拖拽排序。
- 跨日期拖拽。
- 拖到长期重要。
- 拖到长期紧急。
- 从长期分桶拖回日期。
- 滚动列表后继续拖拽。
- 拖拽取消后数据不变。

### 选择和剪贴板

- 点击选择。
- Shift 点击范围选择。
- Option 拖选。
- 复制单个待办。
- 复制多个带层级待办。
- 粘贴到日期列表。
- 粘贴到长期分桶。
- 删除选中待办。

### 撤销重做

- 撤销文本编辑。
- 撤销新建待办。
- 撤销删除待办。
- 撤销缩进变化。
- 撤销拖拽排序。
- 撤销跨日期移动。
- 重做以上操作。

### 多入口同步

- 主窗口编辑后，菜单栏更新。
- 菜单栏编辑后，主窗口更新。
- 跨日期移动后，侧边栏月份仍然正确。

## 10. 主要风险

### 风险 1：重写后重新踩旧坑

处理方式：

- 不重写已有核心排序逻辑。
- 先补测试，再换 UI。
- 每次只迁移一个明确区域。

### 风险 2：AppKit 编辑器和 SwiftData 绑定太深

处理方式：

- 行视图只负责展示和事件。
- 数据变化统一走 `TodoStore`。
- 不让行视图直接管理保存。

### 风险 3：焦点和选择状态变得混乱

处理方式：

- 编辑器内部自己管理临时 UI 状态。
- 对外只暴露菜单、剪贴板、撤销需要的最小状态。
- 不在多个地方重复保存同一份选择状态。

### 风险 4：主窗口和菜单栏行为分裂

处理方式：

- 先明确主窗口编辑器的行为契约。
- 菜单栏后续要么复用同一套编辑器，要么明确保留简化版。

### 风险 5：一次性改太多导致长期不稳定

处理方式：

- 旧 SwiftUI 编辑器保留一段时间。
- 新旧编辑器可以对照验证。
- 每个阶段都必须构建和验收。

## 11. 建议的第一轮实现内容

第一轮实现应该非常小，只证明方向，不碰复杂交互。

建议只做：

1. 新建 `Views/AppKitEditor/`。
2. 新建 AppKit 编辑器入口。
3. 嵌入到 SwiftUI 页面里。
4. 只读展示某一天的待办。
5. 不替换现有正式界面。
6. 不实现编辑、拖拽、多选。
7. 确认构建和现有测试通过。

这一轮完成后，再决定是否进入文本编辑迁移。

## 12. 审阅时需要确认的问题

开始重构前，需要你确认这些方向：

- 第一批迁移只做日期列表，还是日期列表和长期列表一起做？
- 旧 SwiftUI 编辑器是否保留一段时间作为回退？
- 菜单栏列表最终是否也要迁到同一个 AppKit 编辑器？
- 如果 AppKit 的原生体验和当前细节略有不同，哪些行为可以调整？
- 是否接受第一轮只做只读编辑器骨架，不立刻解决拖拽和编辑？
